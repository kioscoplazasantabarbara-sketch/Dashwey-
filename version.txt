CHANGELOG v1.2.34-dev — Fix flujo pedidos: separación currentOrder vs pendientes
==================================================================================

PROBLEMA CORREGIDO
  v1.2.33 mezclaba los roles del botón "Aceptar pedido":
  lo usaba para GUARDAR el pedido (en lugar de MOSTRARLO) y abría el selector
  automáticamente. Los tres puntos de entrada del dock no tenían responsabilidades claras.

FLUJO CORRECTO v1.2.34
  ┌──────────────────────┬──────────────────────────────────────────────┐
  │ Botón/Área           │ Acción                                       │
  ├──────────────────────┼──────────────────────────────────────────────┤
  │ Botón "Pedido" (rojo)│ Abre modal carrito activo (openCurrentOrder) │
  │ Ícono guardar (💾)   │ Guarda carrito → pendingOrder (sin stock)    │
  │ PRÓXIMA ENTREGA      │ 0→toast/informe · 1→check-modal · N→selector │
  │ PEDIDO (col dock)    │ Abre carrito activo (=Botón "Pedido")        │
  │ TOTAL A PAGAR        │ Informe de totales (sin cambio)              │
  └──────────────────────┴──────────────────────────────────────────────┘

[FIX-1] _onAcceptBtnTap — botón "Pedido" → openCurrentOrder()
  - Antes: guardaba el carrito como pendingOrder y abría el selector
  - Ahora: abre el modal del carrito activo (análisis + botón guardar)
  - haptic: 'light' (era 'medium' — apropiado para abrir, no confirmar)

[FEAT-1] openCurrentOrder() — nueva función exportada
  - Carrito vacío → showToast('🛒 No hay artículos en el pedido actual')
  - Carrito con items → openCarritoReport() (modal de análisis existente)

[FEAT-2] openPendingOrders() — nueva función exportada
  - 0 pendientes del proveedor activo → openOrdenReport() (informe clásico)
  - ≥1 pendientes → window._orderSelector.open() (1→directo · N→selector)
  - Lógica 1-vs-N delegada en _orderSelector.open() (ya existente)

[FIX-2] _guardarComoOrdenPendiente — ya NO abre selector automáticamente
  - Antes: abría _orderSelector tras 400ms (flujo confuso)
  - Ahora: guarda silenciosamente y muestra toast + animaciones de feedback
  - El usuario decide cuándo ir a "Próxima Entrega" para recibir el pedido

[FEAT-3] Feedback visual al guardar pedido
  - Flash .alm-dock-save-flash en col PEDIDO (0.55s dockSavePulse)
  - Highlight .alm-dia-highlight en col PRÓXIMA ENTREGA con offset 160ms
    → secuencia: "de aquí (PEDIDO) viene → a aquí (PRÓXIMA ENTREGA) fue"
  - Ambas animaciones solo transform+opacity: GPU friendly, sin layout

[FIX-3] HTML dock — onclick rewired correctamente
  - PRÓXIMA ENTREGA: App.alm.openPendingOrders() (era _onProximaEntregaTap)
  - PEDIDO: App.alm.openCurrentOrder() (era openCarritoReport directamente)
  - TOTAL: App.alm.openTotalReport() (sin cambio)
