FIXES APLICADOS (ordenados por prioridad)
═══════════════════════════════════════════

[C1] CRÍTICO — ALM: rama _ordenEstado===1 era código muerto
     Problema:
       _onAcceptBtnTap() tenía una bifurcación:
         if (_ordenEstado === 1) { _cerrarCatalogo(); setTimeout(..., 300) }
         else { _guardarCarritoYPopup() }
       _ordenEstado NUNCA se asigna a 1 (el estado 1 se eliminó en v8.8.2).
       La rama if nunca ejecutaba. La rama else siempre ejecutaba.
       Ambas ramas liberaban btn.dataset.processing en momentos distintos,
       creando una ventana de inconsistencia en el guard de doble-tap.
     Fix:
       · Rama if(_ordenEstado===1) completamente eliminada
       · Flujo directo: _guardarCarritoYPopup() + processing='0'
       · Código más claro, sin estado zombi

[C3] CRÍTICO — TPV: procesarPago sin try/finally
     Problema:
       Si State.set.addVenta() o State.set.updateProd() lanzaban una excepción
       (localStorage lleno, corrupción de datos, etc.), _procesandoPago quedaba
       en true para siempre. El TPV quedaba completamente bloqueado hasta
       recargar la app — ninguna venta posible.
     Fix:
       · Pasos 3-6 envueltos en try/catch/finally
       · catch: showToast con mensaje de error, no falla silenciosamente
       · finally: _procesandoPago = false — siempre se libera, sin excepción
       · La IIFE de Firebase (paso 5) ya tenía try/catch propio — intacta

[A1] ALTO — ALM: _closeCtx acumulaba listeners en cada longpress
     Problema:
       _openCatInlineCtxMenu() registraba dos listeners en document
       (touchstart + click) para cerrar el menú contextual al tocar fuera.
       Si el usuario cerraba el menú desde un botón interno (Editar/Borrar/etc.),
       el removeEventListener manual NUNCA corría — los listeners se acumulaban
       indefinidamente (memory leak + handlers stale).
     Fix:
       · { once: true } en ambos listeners
       · El listener se auto-elimina tras el primer evento (toque fuera)
       · Si el menú se cierra desde un botón interno, el rAF ya no dispara
         porque el menú está removed del DOM antes del click/touchstart externo

[A6] ALTO — Global: _silentCheckUpdate sin .catch()
     Problema:
       La cadena fetch(version.txt).then().then() sin .catch().
       En modo offline o con red cortada, generaba unhandledRejection
       en consola. No bloqueante pero indicador de error no controlado.
     Fix:
       · .catch(() => null) después del primer .then()
       · Si la red falla → data = null → función retorna silenciosamente
       · Ningún unhandledRejection
